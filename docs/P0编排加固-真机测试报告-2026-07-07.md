## 二、P0 逐项实测

### P0-A 节点心跳注册表（gap P0-3）✅

node-agent 起来后扫 `claude-sbx-nodes` 表：

```
心跳条数: 2
  node_id=ip-10-0-103-204.ec2.internal ip=10.0.103.204 free_mem=123914 vm_count=0 last_seen=2026-07-07T02:42:45Z
  node_id=ip-10-0-101-137.ec2.internal ip=10.0.101.137 free_mem=126304 vm_count=0 last_seen=2026-07-07T02:42:43Z
```

字段完整、last_seen 随周期刷新。**注意**：首次验证时心跳条数为 0，暴露了坑 1（见 §三），修复后通过。

### P0-B leader 选举 + 故障转移（gap P1-4）✅

控制面 2 副本，`claude-sbx-locks` 只有一个 reconciler 锁：

```
owner=sandbox-control-plane-7f8bd98859-p7qmb-634e38 rvn=11 expires=...
```

rvn=11 说明 leader 在持续续租（每 ~10s 一次），无双主。

**故障转移**：删除持锁的 `p7qmb` pod →

```
新 owner=sandbox-control-plane-7f8bd98859-722hn-dce2f4 rvn=15
```

存活副本 `722hn` 接管锁、rvn 继续自增。故障转移成立。

### P0-C 节点发现（gap P0-3）✅

create 一个沙盒，返回落点：

```
state=running node=10.0.101.137 tap_idx=14 guest_ip=172.18.14.2
```

`node` 是心跳表里的活节点 IP，证明 `_pick_node` 从注册表选点（非环境变量 FC_NODES）。exec 走 vsock 验证确在 microVM 内：

```
exec {echo sandbox-ok && uname -r && nproc} → rc=0, stdout="sandbox-ok\n5.10.223\n1"
```

guest kernel 5.10.223 ≠ 宿主 6.1.x、nproc=1=guest 配额 → 确在 microVM 内执行。

### P0-D S3 上传强一致（gap P0-2）✅

写标记数据 `P0-MARKER-42` → suspend：

```
suspend → state=suspended  snapshot_s3=s3://.../sbx/d87e3e8a/  size=536870912

# S3 实有内容:
2147483648 rootfs.ext4
 536870912 vm.mem
      6308 vm.snapshot
```

**不变式成立**：标 `suspended` 时 S3 确有完整三件套。这正是本次改动的核心——旧代码 fire-and-forget 异步上传、标 suspended 时 S3 可能还没传完；新代码同步确认上传成功才释放 VMM 内存。

resume + 数据保真：

```
resume → state=running restore_time_s=0.1136
exec {cat /tmp/marker} → rc=0, stdout="P0-MARKER-42"
```

resume **0.11s**（亚秒级），标记文件完整恢复（内存态跨快照精确保留）。

### P0-E reconcile 漂移检测（gap P0-1）✅

构造漂移：向 sandboxes 表插入一条 `state=running`、`node=<活节点>`、但节点上无此 VM 的记录 `drift-test`，等 reconcile 一轮（20s 周期）：

```
drift-test → state=orphaned  reconcile_reason=runtime_unknown
```

reconcile 检出 DynamoDB 说 running 但 `get_runtime_state` 返回 unknown 的漂移，自动标 orphaned。

**补充验证（reconcile 兜底真实场景）**：删掉某沙盒所在节点的 node-agent pod（模拟 node-agent 重启），其 FC 子进程随 pod 死亡；`_recover_vms` 探测 socket 已 dead、正确地不恢复；30s 后 reconcile 把该沙盒标 orphaned（runtime_unknown）。证明即便 `_recover_vms` 救不回 VM，reconcile 也保证状态不永久漂移。

## 三、发现的真实问题（mock 测不出，只有真机暴露）

### 坑 1 — node-agent 镜像里 boto3 不可用 → 心跳静默失败

**现象**：node-agent pod Running 但 nodes 表始终为空。
**根因**：心跳代码原用 `import boto3`，但容器里 `python3` 是 3.13（`aws` cli 自带），boto3 未装在该解释器 → `ModuleNotFoundError`，被心跳循环的 `except: pass` 静默吞掉。
**修复**：心跳改用 `aws dynamodb put-item` 子进程（与 S3 快照上传同款 aws-cli 依赖，不依赖 boto3）；并把静默 `except` 改成打 stderr 便于排障。重建镜像 + 重启 DaemonSet 后心跳恢复。

### 坑 2 — 容器化 node-agent 下 `_recover_vms` 对 running VM 无效

**现象**：删 node-agent pod 后自恢复没救回存活沙盒。
**根因**：node-agent 用 `subprocess.Popen` 起 FC 进程，是 pod 容器的子进程；pod 一死，FC 子进程随容器 PID namespace 一起被杀。socket 文件残留但进程没了 → `_recover_vms` 探测连不上、正确地不恢复。
**结论**：这是符合预期的行为，非 bug。`_recover_vms` 只在"node-agent 崩溃重启但 FC 进程存活"场景有用（宿主 systemd 部署，或 pod 用 shareProcessNamespace + 进程托管）；容器化默认部署下 VM 随 pod 死，靠 reconcile 标 orphaned 兜底。这是待写进设计文档的架构边界。

### 坑 3 — c6g.metal 节点反复 NotReady / ASG 替换死循环

**现象**：metal 节点起来后反复 NotReady↔Ready，ASG 每 ~6min 替换一轮，30min+ 收敛不到全 Ready。
**根因**：ASG health check grace period 默认 **15 秒**，而 c6g.metal 过 EC2 status check（system+instance reachability）需 **5-10 分钟** → 节点刚起 15s 就被判 unhealthy 替换。ASG scaling activity cause 明确写 "EC2 instance status checks failure"。**这纠正了旧认知**（曾把此现象归为"metal 冷启动暂态，耐心等自愈"）。
**修复**：把 grace period 调到 900s（`aws autoscaling update-auto-scaling-group --health-check-grace-period 900`）。已在 `terraform/phase3/main.tf` 用 `null_resource.metal_asg_grace_period` 固化（EKS 托管节点组的模块/API 不暴露该参数，只能创建后 patch）。

### 附带 — EKS endpoint 白名单锁 /32 遇 NAT 池浮动

出口 IP（NAT 池）从 `.110` 浮到 `.100` 后 kubectl 突然 TLS handshake timeout。已把 phase3 的 `endpoint_public_access_cidrs` 固化为 `5.34.218.0/24` 覆盖整个池。

## 五、遗留 / 未验（如实记录）

- **死节点 suspended → needs_reschedule 的自动重调度**：本次只验证了标记 `needs_reschedule`（reconcile 打标），未验证自动跨机 restore——按 P0 边界，自动重调度留 P1（需 fencing 防脑裂）。
- **`_recover_vms` 在宿主 systemd 部署下的有效性**：本次是容器化部署，FC 随 pod 死，未能正向验证自恢复救回 VM 的路径。
- **大规模 reconcile 性能**：`list_by_states` 目前全表 scan，POC 规模（数百沙盒）可接受，量大需建 state-index GSI（已在代码注释标注）。
- **leader fencing token 尚未接到状态写路径**：rvn 已产出，作为 P1 自动重调度的预留，本批状态写仍靠 prev_state 条件写防并发。

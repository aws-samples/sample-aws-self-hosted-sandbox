> 集群：EKS 1.31 `claude-sbx` + 1×`c6g.metal` 系统节点 + Karpenter 拉 1×`c6g.metal` kata 节点
> 验证方式：真机部署，同一套控制面代码，`SANDBOX_DRIVER` 环境变量切换 driver，全程 kubectl/curl 实测

**一句话**：
- **Kata**：生命周期管理完整可靠，快照按设计返回 501（QMP socket 被 kata-runtime 独占），exec 有一个时序竞态 bug。
- **Firecracker**：suspend/resume 机制真实可用（**resume 实测 58ms**），但在无本地 NVMe 的 `c6g.metal`（只能用 EBS gp3）上 suspend 超时；exec 因**容器/rootfs 三重缺失**完全不 work。

## 三、发现的真实缺陷

### 坑 1 — create 乐观写 running，导致紧接着 exec 打空

`app.py:159-160`：
```python
driver_fields = _driver.create(sid, spec)
db.force_update(sid, {**driver_fields, "state": "running"})   # 不等 pod/VM 真就绪
```

- Kata `create()` 只提交 pod manifest 就返回，控制面立即标 `running`。
- 实测：create 返回时 `state=running`，但 `kubectl get pod` 显示 `ContainerCreating`。
- 此时 `kubectl exec` stream 对未就绪容器返回 None → `'NoneType'.decode` 报错。
- **等容器真 ready（~几秒）后 exec 完全正常**（实测 rc=0）。

**影响**：客户端"创建后立刻 exec"会偶发 500。`wait?state=running` 因为是乐观标记也会秒过，无法作为真正就绪的信号。
**建议**：`get_runtime_state` 已能查真实 Pod phase + readiness，create 后应轮询到 ContainersReady 再标 running。

## 四、Firecracker exec —— 不 work（三重缺失）

`SANDBOX_DRIVER=firecracker`，`GET /capabilities` → `{"suspend_resume": true, "warm_pool": true, "migrate": true}`

**create 是成功的**：T3 create → 201，T8 locate 返回 `guest_ip: 172.18.24.2`、`node: 10.0.102.103`、`tap_idx`。vm.log 坐实 microVM 完整启动：
```
Run /sbin/sbxinit as init process
[sbxinit] ready
root@(none):/#     ← guest bash 起来了
```

**但 T9 exec 失败**，HTTP 500：`node-agent POST /vm/exec → 500: [Errno 2] No such file or directory: 'ssh'`

三重缺失（逐层坐实）：

| 层 | 缺失 | 证据 |
|---|---|---|
| ① node-agent 容器 | **无 `ssh` 客户端、无 `socat`** | 容器内 `which ssh socat` 全空；`op_exec` 调 `subprocess.run(["ssh",...])` 直接 FileNotFound |
| ② rootfs（guest） | **无 sshd 服务端** | rootfs tar 内 `grep /sshd$\|dropbear` 落空，仅有 ssh 客户端 + sbxinit |
| ③ rootfs（guest） | **无 vsock listener（socat）** | rootfs 内无 socat；vsock 兜底通道 guest 端无监听者 |

`node-agent/main.py` `op_exec` 逻辑是 **TAP SSH 优先 → vsock+socat 兜底**。三层缺失导致两条通道都走不通。

**根因**：FC 沙盒不是 K8s Pod，没有 kubelet 代理进程；exec 依赖 rootfs 内自备 sshd 或 vsock 服务端 + node-agent 容器内有对应客户端。当前预 build rootfs 和 node-agent 镜像都没准备这些。
**结论**：FC 下 exec **当前不 work**。修复需：(a) node-agent 镜像装 openssh-client/socat；(b) rootfs 装 sshd 或 vsock listener 并在 sbxinit 启动。

## 五、Firecracker suspend/resume —— 机制 work，gp3 上 suspend 超时

### resume：✅ 实测成功，58ms

从完整快照产物恢复（先 kill 原 VM，符合 FC "禁止 original+clone 同跑"）：
```
resume OK after 0.14s: {"restore_time_s": 0.0578, "ip": "172.18.29.2"}
```
resume 后 `GET /vm/<id>` → `state=running, pid=36879`，VM 真正复活。**亚秒级恢复坐实（58ms）**，这就是"24×7 可达 ≠ 24×7 常驻"的成本核心杠杆。

### suspend：⚠️ 快照本身成功，但整体超时

Firecracker snapshot API **完全正常**（vm.log 坐实）：
```
'pause vm' VMM action took 52 us
'create full snapshot' VMM action took 105_756_040 us   ← 105 秒
```
产物正确生成：`vm.mem`(2GB) + `vm.snapshot`(6KB) + base 副本。

**但整个 `/suspend` 超时**（控制面 120s / 手动 150s 都超）。分解耗时：

| 步骤 | 耗时 | 瓶颈 |
|---|---|---|
| pause VM | 52µs | — |
| Full 快照写 2GB 内存 | **105s** | EBS gp3 |
| `shutil.copy2` 复制 2GB base | ~25s+ | EBS gp3（实测 base 文件仍在增长时超时） |
| cp 6GB rootfs + kill | 未走到 | — |

**根因 = 磁盘性能**：
- 节点根盘是 **EBS gp3**（`nvme0n1`，NVMe 接口但网络 EBS），`/var/lib/sbx` 在其上。
- 实测直写速度 **81.6 MB/s**（`dd bs=1M count=1024 oflag=direct`）。
- 写 2GB 快照 + 复制 2GB base 累计远超 API timeout。
- `c6g.metal` **没有本地 NVMe instance store**，只能用 EBS → 快照注定慢。

这与 POC 文档 §二点五 早期预警完全吻合：*"Full 快照把内存全量写 EBS gp3，可优化：写本地 NVMe（`i` 系列 .metal）数量级提升"*。历史 POC 记的 33s 是 2GB 但更快的 gp3/更少叠加；本次多了 `shutil.copy2` base 复制的叠加。

**结论**：FC suspend **机制正确、产物正确、resume 可用**，但在 `c6g.metal`(EBS gp3) 上 **suspend 因磁盘慢超出 API timeout → e2e 视角失败**。
**修复方向**：(a) 用带本地 NVMe 的机型（`i4g`/`im4gn` 等）承载快照；(b) diff 快照只写脏页；(c) 加大/异步化 suspend timeout，快照与 base 复制解耦。

## 六、两 Driver 定位总结

| 维度 | Kata (QEMU) | Firecracker |
|---|---|---|
| 适用场景 | 标准 K8s 生态、exec 走 kubelet、无需快照 | 需要 suspend/resume 成本优化、亚秒恢复 |
| exec 现状 | 可用（有就绪竞态待修） | 不可用（需补 sshd/socat + 客户端） |
| 快照 | 不支持（QMP 被 kata-runtime 独占，设计返回 501） | 支持，resume 58ms；suspend 需本地 NVMe 机型 |
| 真 microVM | ✅ | ✅ |
| 部署复杂度 | 中（Karpenter bootstrap 预装 kata） | 高（rootfs/kernel/FC 二进制/tap 网络全走 node-agent） |

## 七、本次部署中排除的真实问题（供复现参考）

1. **EKS apply 被 10min shell 上限 SIGTERM 中断** → cluster 创建但未落 state → 重跑报 409。修复：`terraform import module.eks.aws_eks_cluster.this[0] claude-sbx`，且 import 读回的 `bootstrap_self_managed_addons=false` 是 ForceNew 假值，需手改 state 为 true 才不触发 cluster 重建。
2. **IP 漂移**（同段尾号变化）导致 endpoint CIDR 反复挡住 kubectl → 锁 `/24` 出口段解决。
3. **FC 资产与 node-agent 落点错配**：装 FC 资产的系统节点默认无 `sandbox=true` label，node-agent 只调度到该 label → 手动补 label。
4. **系统节点 userData 未执行**（`/var/log/userdata-pre.log` 不存在）→ FC 资产缺失 → 手动在宿主 nsenter 安装 firecracker/kernel + kubectl cp rootfs。
5. **node-agent 容器无 `/opt/sbx` 挂载** → FC 资产放到已挂载的 `/var/lib/sbx` 并改 `FC_ROOTFS`/`FC_KERNEL_PATH` 指过去。
6. **控制面 auth 强制开启**，e2e 脚本需带 `Authorization: Bearer <API_KEY>`（原 `scripts/e2e_test.sh` 未带，会全 401）。

## 八、修复与复测（2026-07-01 同日）

针对 §四（FC exec 不 work）和 §五（FC suspend 超时），做了根因修复并真机复测通过。**Firecracker e2e 从 10/12 → 14/14 PASS。**

### 8.1 FC exec 修复 —— vsock 通道（绕开 SSH + sbxinit IP bug）

选用 vsock 而非 SSH：不依赖 guest 网络配置，避开了 sbxinit 硬编码 IP 的坑（sbxinit 写死 `172.18.1.2`，但实际 tap_idx 动态，两者只有 tap_idx=1 时才一致——这是 SSH 通道即使装了 sshd 也连不上的深层原因）。

改动三处：
1. **guest 端**（`scripts/vsock-exec-agent.py`，新增）：Python 写的 vsock listener，监听 `AF_VSOCK` port 2222，收 JSON 请求 → `bash -c` 执行 → 回 JSON。利用 rootfs 里已有的 python3.11，无需装 sshd/socat。
2. **rootfs**：sbxinit 在 `[sbxinit] ready` 后后台启动该 agent。
3. **node-agent**（`node-agent/main.py` `op_exec`/`_vsock_exec`）：实现 Firecracker vsock 的 `CONNECT 2222\n` 握手协议（原代码用 `socat UNIX-CONNECT` 缺握手且容器无 socat），改用纯 Python socket；通道优先级调整为 vsock 优先、SSH 兜底；并对"create 乐观 running 但 guest 未就绪"加了最多 ~10s 重试。

**复测**：`exec {echo sandbox-ok && uname -r && nproc}` → `rc=0, stdout="sandbox-ok\n5.10.223\n1"`（guest kernel + guest 配额，证明在 microVM 内执行）。非零退出码、stderr、连续多次调用、写文件读回均验证正确。

### 8.2 FC suspend/resume 修复 —— 三个真 bug

复测中发现 §五的 timeout 背后其实叠了三个 bug，逐个修复：

1. **`shutil.copy2` 复制 2GB base 阻塞**（gp3 ~81MB/s → ~25s）：改 `cp --reflink=auto`（ext4 CoW，base 复制瞬间完成、不占额外空间）。
2. **node-agent 容器无 `/bin/kill`**（python:3.12-slim 无 util-linux）：`subprocess.run(["kill",...])` 报 `No such file: 'kill'` → 改 `os.kill(pid, signal.SIGTERM)`。**这个 bug 还连带解释了 §七的磁盘泄漏**：op_destroy 里同样的 kill 失败抛异常，导致后面的 `shutil.rmtree` 清理没执行 → 沙盒目录残留（每个 6GB+2GB）→ 批量测试撑爆 200GB 盘触发 DiskPressure 驱逐 pod。修 kill 后 destroy 能正常清盘。
3. **resume 时 vsock UDS "Address in use"**：快照本身含 vsock 设备配置（含 host UDS 路径 `v.sock`），load 时 FC 自动重建并绑定该 UDS，但旧 `v.sock` 文件残留导致 `bind: Address in use (os error 98)` → resume 报 400。修复：load 前 `os.remove(v.sock)`。
4. 控制面 `_agent` 的 urlopen timeout 参数化，suspend 用 300s（gp3 上 Full 快照可能 ~90s）。

**复测（完整冻结-解冻语义验证）**：
```
create → exec 写 /run/marker=MAGIC_42（tmpfs 纯内存）
       → suspend (state=suspended, 快照 22~94s，随 gp3 IO 竞争波动)
       → resume  (state=running, restore_time 0.069~0.105s)
       → exec: cat /run/marker → "MAGIC_42"  ← 内存状态精确跨快照保留
```
resume 恢复延迟 **69~105ms**，内存态（tmpfs 里的 MAGIC_42）完整保留，证明是真正的冻结-解冻而非重启。

### 8.4 遗留 / 未修（如实记录）

- **suspend 耗时不稳定（22~94s）**：根因仍是 `c6g.metal` 只能用 EBS gp3（无本地 NVMe），Full 快照写 2GB 内存受 ~81MB/s 带宽限制，且与同节点其它 VM 抢 IO。已用 reflink + 300s timeout 让它"能完成"，但要"快且稳"需换带本地 NVMe 的机型（`i4g`/`im4gn`）或启用 diff 快照。
- **create 乐观写 running（kata & FC 共有）**：`app.py` create 后立即标 running 不等真就绪。FC 侧已在 exec 层加重试兜底；根治应让 create/wait 反映真实就绪（kata 查 pod ContainersReady、FC 探测 vsock agent）。本次未改 create 语义以免影响暖池等逻辑。
- **sbxinit 硬编码 guest IP**：`172.18.1.2` 写死，SSH 通道对 tap_idx≠1 的沙盒本就连不上。因已改用 vsock 通道，SSH 仅作兜底，未修此 bug。

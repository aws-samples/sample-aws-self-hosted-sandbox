## 零、结论速览

**架构定位**：我们这套「中心化 control plane + DynamoDB 状态 + S3 快照 + 跨机 restore」在业界最像 **E2B**（`e2b-dev/infra`：Firecracker + Nomad + Consul + envd）。E2B 基本就是"用 Go 写好、并补齐了我们缺口的 Python control plane"，是最直接的对标对象。Fly.io 的去中心化 gossip/CRDT（Corrosion）对我们**单区 AWS + DynamoDB** 属过度设计，只借它两个理念（proxy 驱动 autostop、pinned vs reschedule 权衡）。

**战略方向已定**：加固现有 Python control plane —— **不上** K8s CRD / kopf / KubeVirt，**不换**语言。我们缺的四块（reconcile / 节点发现 / leader 选举 / autostop）恰好是 K8s 控制面天生解决的问题，但我们选择用 **DynamoDB 原语在 Python 内自建等价能力**。

**短板 → 业界解法 → 我们的实现配方** 映射表：

| 短板 | K8s 原生（Kata 白嫖） | E2B（Go）做法 | 我们（Python + DynamoDB）照抄的配方 |
|---|---|---|---|
| **reconcile loop** | kubelet → etcd 持续对账 | api 侧 ~20s sync loop（`keepInSync → syncNodes → syncNode`） | level-triggered 后台 loop 对账 DynamoDB vs node-agent 实况 |
| **节点发现** | Node 对象 + kubelet 自注册 | Nomad/Consul + Redis catalog | DynamoDB `nodes` 心跳表（`last_seen` TTL 剔除死节点） |
| **leader 选举** | controller-runtime Lease（一个开关） | —（Nomad 托管） | DynamoDB 条件写租约 + fencing token（AWS DynamoDB Lock Client 模式） |
| **autostop** | —（需自建） | 默认 5min idle → pause | proxy 唤醒 + reconcile loop 收割（Fly 式拆分）+ `last_activity` |
| **调度 / 放置** | kube-scheduler | 自研 Best-of-K 装箱（**故意不用** Nomad 调度器） | power-of-K-choices：采样 K 个节点按 CPU 超卖 + 近期负载打分 |

## 一、外部方案对标（只提炼编排层）

### 1.1 E2B — 架构双胞胎，重点最多

开源仓库 `github.com/e2b-dev/infra`，Go monorepo。它和我们的对应关系几乎逐模块成立：

- **中心化 control plane（Go）**：外部走 OpenAPI/HTTP（`packages/api`），control-plane → node 走 gRPC，SDK → guest agent（`envd`）走 Connect RPC。↔ 我们的 `sandbox-api/app.py` + node-agent HTTP。
- **自研放置算法，故意不用 Nomad 调度器**：placement 是 api 里的 "Best-of-K" 装箱器（`packages/api/internal/orchestrator/placement/`）——采样 K=3 个候选节点，按 `(cpuRequested + reserved + Alpha·usageAvg) / (R · cpuCount)` 打分（超卖比 R=4、Alpha=0.5），选最低分，支持 `preferredNode`（resume 回温节点）、失败重试、排除故障节点。**Nomad 只放置平台服务，沙盒放置是 E2B 自己的代码。** ↔ 我们的 `drivers/firecracker.py:_pick_node`（现在只看 free_mem）。
- **快照进对象存储（S3/GCS）**：真·Firecracker 内存快照（memory diff + rootfs diff + snapfile + metafile，链式 diff），`STORAGE_PROVIDER` 选 S3。因为快照在中心存储 + resume 重跑 placement，所以 **resume 可跨节点**。resume ~1s 靠 **UFFD 惰性分页**（按 page-fault 拉内存，不整块拷）。↔ 我们的 S3 快照三件套 + 跨机 restore（已真机验证，commit be02043）。
- **~20s reconcile sync loop**：`keepInSync → syncNodes → syncNode` 周期对账 api 侧沙盒存量 vs 每节点上报；gRPC 失败标节点 Unhealthy；节点状态机 **Ready / Draining / Unhealthy / Standby**。**运行中 VM 不迁移**——节点脏死则其运行沙盒丢失，只有 **paused（快照在对象存储）能跨机 resume**。↔ 正是我们缺的 reconcile（gap P0-1）。
- 来源：`packages/api/internal/orchestrator/placement/placement_best_of_K.go`、`.../nodemanager/sync.go`、`.../cache.go`、e2b.dev/docs/sandbox/persistence、e2b.dev/blog/up-to-5x-faster-sandboxes。

> **一句话**：E2B 是我们的"未来态参考实现"——它把我们缺的 reconcile / 放置算法 / 快照分层 / 节点状态机全部写好了，只是用 Go。逐模块对标它的 `api/internal/orchestrator` 包即可。

### 1.2 Fly.io（flyd / Corrosion / fly-proxy）— 只借 2 个理念

- **去中心化**：每台 worker 一个 `flyd`，各自是自己 VM 的唯一事实源；状态存**本机 BoltDB append-only log**，崩溃后从日志续跑（事件溯源，不跑周期 diff）。`flaps`（Machines API）扇出到各 worker。
- **Corrosion**：Rust 写的 gossip（SWIM）+ CRDT（cr-sqlite，逻辑时钟 LWW）同步全局路由表，替代 Consul 中心目录；`fly-proxy` 读本机 SQLite 就近路由到目标 machine。
- **对我们的相关性**：**gossip/CRDT 这套是过度设计**——Fly 需要它是因为跨洲多区、"共识协议跨长距离会崩"；我们单区 AWS + DynamoDB 强一致完全够用。**只借 2 个理念**：
  1. **proxy 驱动 autostop/autostart**（scale-to-zero）：请求打到已停 machine 时由 proxy 唤醒，后台 loop 收割空闲 —— 见配方 4。
  2. **pinned vs reschedule 权衡**：Fly 的快照在本机 → machine pinned、不能自动迁移，只能预置 standby；我们快照在 S3 → **可以做 reschedule**，这是我们相对 Fly 的优势。
- 来源：fly.io/blog/carving-the-scheduler-out-of-our-orchestrator/、fly.io/blog/corrosion/、fly.io/docs/reference/fly-proxy-autostop-autostart/。

### 1.3 Kubernetes 原生 / Kata — 我们 Kata driver 白嫖的能力，及其天花板

Kata 沙盒就是**普通 Pod**（`runtimeClassName: kata-qemu`），因此**免费获得**：kube-scheduler 调度、Deployment/ReplicaSet 控制器的 reconcile 自愈、liveness/readiness 探针、HPA、节点 NotReady 自动驱逐 + 重建。**这正是我们 Kata driver 不需要手搓、而 FC driver 必须自建的那一整圈运维层。**

**天花板（K8s 不给的，也是 FC 路线的存在理由）**：
- Kata runtime **无 checkpoint/restore**，VM save/restore 官方标"under discussion" → **内存快照、suspend/resume、亚秒 restore、跨机有态恢复，Kata 标准路径不提供**。
- K8s 把 Pod 当易失/无状态：**Pod（按 UID）永不迁移**，节点故障是删旧建新、运行态不保留。
- 来源：kata-containers `docs/Limitations.md`、kubernetes.io/docs/concepts/workloads/pods/pod-lifecycle/。

### 1.4 KubeVirt / Virtink — 为什么我们不走 CRD 路线

把 VM 做成 CRD（VMI/VM controller）是"把 VM 纳入 reconcile 循环"的标准答案，但对我们**不适配**：

- **都不跑 Firecracker**：KubeVirt = libvirt/QEMU/KVM，Virtink = Cloud Hypervisor。
- **跨机模型是 live-migration**（pre-copy 内存拷贝，要求 **RWX 共享存储** + 网络传内存），**不是**我们的"快照到 S3 + 跨机 restore"。KubeVirt 的 `VirtualMachineSnapshot` 是**磁盘卷快照，非内存运行态检查点**，恢复通常要求 VM 停止。
- 结论：我们的 FC 内存快照跨机 restore **落在 KubeVirt 模型之外**，走 CRD 反而要放弃差异化能力。仅作对照，说明"为何加固 Python 而不迁 CRD"。
- 来源：kubevirt.io/user-guide/compute/live_migration/、kubevirt.io/user-guide/storage/snapshot_restore_api/、github.com/smartxworks/virtink。

### 1.5 CodeSandbox — 内存超卖配方，且我们在跨机上已平齐/领先

FC 内存快照的工程细节最详实的一手来源：
- 快照 = 暂停 VM + 写 `snapshot.snap`（vCPU/设备）+ `memory.snap`（全内存）；`MAP_SHARED` mmap 把写延迟从 8–12s 砍到 30–100ms；resume ~0.6s 靠 mmap + page-fault 惰性载入。
- **内存超卖配方**：快照按 **8KiB 分块压缩**（zstd ~20× / lz4 ~10.7×）、page-fault 时按需解压（userfaultfd handler）、零页存 0 字节、virtio-balloon **Free Page Reporting** 排除已释放页。
- **关键**：CodeSandbox 的 **live 跨机迁移仍是 future work**（截至 2024）——**我们已真机验证的跨机 restore（commit be02043）在这一点上已平齐甚至领先**。
- 来源：codesandbox.io/blog/how-we-clone-a-running-vm-in-2-seconds、codesandbox.stream/blog/how-we-scale-our-microvm-infrastructure-using-low-latency-memory-decompression。

### 1.6 其余平台 — 一表带过

| 平台 | 隔离 | 控制面 | Autostop | 快照/resume |
|---|---|---|---|---|
| **Modal** | gVisor (runsc) | 自研 Rust + GLOP 求解器 | `scaledown_window` 默认 60s | gVisor checkpoint/restore |
| **Daytona**（开源） | namespaces（容器级） | NestJS + PG + Redis，有 reconcile | SDK `auto_stop_interval` | OCI 镜像快照 → S3 |
| **Runloop** | 隔离 VM | — | 默认 1h，`after_idle` suspend/shutdown | 磁盘快照 + suspend/resume |
| **Blaxel** | 快启 VM | — | ~15s 无活动 → 快照整机到 standby | 全状态快照 |
| **Northflank** | Kata/gVisor + FC | — | microVM <1s 启动 + bin-packing | — |

**规律**：控制面几乎都自研或用非 K8s 编排器（无人把 FC 做成 KubeVirt CRD）、**autostop 是标配**、**内存快照 pause/resume 是护城河**（我们已具备）。

### 1.7 OpenSandbox（用户点名）— 借组件分层，不借快照

`opensandbox-group/OpenSandbox`：FastAPI（Python 44%）+ Go（30.8%）+ K8s，隔离支持容器 / gVisor / Kata / Firecracker 可选。组件分层清晰：

- `server/`（FastAPI 控制面）、`components/execd`（节点执行守护）、`components/ingress`（统一入口网关 + 多路由策略）、`components/egress`（每沙盒出口控制）、凭证库（安全注入，不暴露真实密钥）、`sdks/`（多语言）。
- **未提供** suspend/resume / 快照 / 跨机迁移（README 未提及）。重点在**网络隔离（ingress/egress）+ 凭证注入**。
- **可借鉴的是组件分层**：`execd`（≈我们 node-agent）/ `ingress`（≈我们缺的 sandbox→node 路由，见 P2）/ `egress`（每沙盒出口策略，我们手搓 iptables 的升级方向）/ 凭证库（多租户密钥注入）——**而非**编排/快照能力（那块它不如我们）。
- 来源：github.com/opensandbox-group/OpenSandbox。

## 二、四块短板的实现配方（本文档核心）

每块给：**业界模式来源 → 我们用什么 DynamoDB 原语实现 → 复用现有哪段代码 → 注意事项**。

### 配方 1 — level-triggered reconcile loop

**来源**：K8s controller 的 level-triggered 模型（每 tick 重算 desired vs observed，丢消息自愈）+ E2B 的 ~20s sync loop。

**实现**：
- 控制面加后台 loop，周期扫 `running / suspended / suspending / resuming` 记录 → 逐个调 **`driver.get_runtime_state`**（`driver.py:74` **已定义、当前无人调用**）对账 node-agent 实况 → 漂移则修正状态或告警。
- node-agent 侧加**启动自恢复**：扫 `SBX_BASE` 下残留目录 + 探测 FC `api.sock` 存活，重建 `_VMS`（`node-agent/main.py:64` 的纯内存 dict 是状态漂移根因——重启即丢句柄）。
- 所有对账动作**幂等**；level-triggered（重新断言而非依赖事件），一次丢包下 tick 自愈。

**注意**：reconcile loop 本身必须**只在 leader 副本跑**（依赖配方 3），否则多副本重复对账互相打架。对应 **gap P0-1**。

### 配方 2 — DynamoDB 心跳节点注册表

**来源**：E2B 的节点存量 + 状态机；K8s 的 Node 对象 + kubelet 自注册。我们**不上 gossip**（Fly 的 Corrosion 只在跨洲多区高 churn 才划算，见 §1.2）。

**实现**：
- 新增 DynamoDB `nodes` 表：node-agent 定期心跳 upsert `{node_id, ip, free_mem, vm_count, last_seen, labels}`。写法参照 `db.alloc_tap_idx`（`db.py:206`）的原子 upsert。
- 控制面从表里读活节点，`last_seen` 超时（如 3× 心跳周期）即视为死、剔除出调度池 —— 替换 `drivers/firecracker.py:174` 的 `FC_NODES` 环境变量硬编码。
- 与配方 1 结合：死节点上的沙盒标 `needs_reschedule`，用**已验证的 S3 快照跨机 restore** 在健康节点重建。

**注意**：**restore 前必须 fencing**（用配方 3 的 fencing token 或对沙盒记录条件写"锁死旧节点"），否则旧节点假死复活会与新节点双开同一沙盒 → 脑裂。接受 RPO = 上次快照（运行中未快照的 VM 仍会丢，需要更紧 RPO 则周期后台快照）。对应 **gap P0-3**。

### 配方 3 — DynamoDB 条件写 leader 选举

**来源**：AWS DynamoDB Lock Client 模式（条件写租约 + Record-Version-Number 做 fencing token）。

**实现**：
- 单 lock item（如 `id = "lock:reconciler"`）字段 `{owner, rvn, expires}`。抢锁/续租用 `ConditionExpression`（`attribute_not_exists(owner) OR expires < :now OR owner = :me`），续租周期 = TTL/3，租约时长 > 最大 GC 停顿。
- 每次状态变更写带上 `rvn` 作 **fencing token**，防"暂停后复活的旧 leader"用陈旧视图乱写。
- 后台 loop（reconcile、暖池补充 `warm_pool.py:117`、autostop 收割）**只在持锁副本跑**；无状态请求路径仍多副本。

**注意**：现有 `db.acquire_lease`（`db.py:140`）是 **per-sandbox 60s 乐观锁**（防并发操作同一沙盒），**不是 leader 锁**——需**新增独立 lock 逻辑**，但可完全复用它的"条件写 + `lease_expires` TTL 过期"模式（`db.py:144-149`）。对应 **gap P1-4**（多副本暖池打架 / 单副本单点）。

### 配方 4 — autostop / idle 检测（Fly 式拆分）

**来源**：Fly 的"proxy 唤醒 + 后台 loop 收割"拆分；E2B 默认 5min idle → pause。

**实现**：
- DynamoDB 沙盒记录加 `last_activity`，每次 exec / 连接刷新。
- **收割**（reconcile loop 内，leader-only）：`now - last_activity > idle_threshold` → FC 触发内存快照 suspend / Kata 停 Pod。默认阈值参照 E2B **5min**，可按 `ServiceSpec.autostop` 覆盖。
- **唤醒**：流量打到已 suspend 沙盒时，由请求路径 **auto-resume**（复用现有跨机 restore + 暖池 claim 的亚秒恢复）。
- 落地 `driver.py:19` 的 `ServiceSpec.autostop` 占位字段（当前只是注释"下阶段实现"）。

**注意**：suspend 是"省钱"卖点的**触发器**，但依赖 S3 上传可靠（见路线图 P0 的 S3 项），否则 idle 自动 suspend 会放大 gap P0-2 的静默丢数据风险。对应 **gap P1 autostop**。

## 三、分阶段路线图

对齐 [`fc-生产就绪-gap分析.md`](./fc-生产就绪-gap分析.md) §五的推进顺序，补上"用哪个配方"。

### P0 — 真实流量前必堵（防静默数据丢失 / 状态漂移）

| 项 | 配方 | gap 锚点 |
|---|---|---|
| **leader 选举**（其余 P0 的前置依赖） | 配方 3 | gap P1-4 |
| **reconcile loop** | 配方 1 | gap P0-1（`node-agent/main.py:64`、`driver.py:74`） |
| **节点心跳注册表** | 配方 2 | gap P0-3（`drivers/firecracker.py:174`） |
| **S3 快照上传可靠性** | —（直接改现有代码） | gap P0-2：上传**成功确认后**才置 `suspended`；上传中用中间态；失败重试+回滚+告警；**绝不在确认前 kill VMM**（`node-agent/main.py:238-250`） |

> 顺序建议：**leader 选举 → reconcile loop → 节点注册表 → S3 可靠性**。前三个有依赖链，S3 可独立并行。

### P1 — 规模化必需

- **autostop / idle**（配方 4）——FC 相对 Kata 的核心价值主张，不做则"省钱"停在手动。
- **Best-of-K 多维调度**——用 power-of-K-choices（采样 K 个节点，CPU 超卖 + 近期负载 + vm_count 打分）替换 `firecracker.py:150-167` 只看 `free_mem`；参照 E2B `placement_best_of_K.go`。
- **磁盘水位 + 快照 GC**——磁盘配额 + 孤儿目录清理（结合 reconcile）+ S3 生命周期策略 GC（gap P1，已撞过撑爆 200GB）。
- **suspend 异步化 + 背压**——提交任务 + 轮询状态，加并发上限，防 ThreadingHTTPServer 线程耗尽（gap P1）。

### P2 — 对齐 Kata 功能完整度

- sandbox→node **路由 catalog**（借 OpenSandbox `ingress` / E2B client-proxy 模式）。
- 就绪探针（替换 create 乐观写 running，gap P2）。
- jailer 默认开启（`USE_BARE_FC=1` → 生产切 jailer，gap P2）。
- 可观测性：metrics / tracing / 结构化日志（当前日志被静音，gap P2）。
- 出口策略（借 OpenSandbox `egress` 升级手搓 iptables）、RBAC / 审计。

## 四、明确不做（划清边界，避免过度工程）

- **不上** K8s CRD / kopf / KubeVirt / Operator：用户已定方向；且 KubeVirt/Virtink 不跑 Firecracker、跨机是 live-migration 而非我们的快照 restore（§1.4）。
- **不引入** Fly 式 gossip / CRDT（Corrosion）：单区 AWS + DynamoDB 强一致已够；gossip 只在跨洲多区高 churn 才划算（§1.2）。
- **不做** VM live-migration（运行态跨机热迁移）：走"快照到 S3 + 跨机 restore"路线，接受 RPO = 上次快照。这也是我们相对 Fly（pinned）的差异化优势。

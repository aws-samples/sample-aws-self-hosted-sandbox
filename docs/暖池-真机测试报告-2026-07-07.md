# 暖池（Warm Pool）真机 e2e 测试报告

- **日期**：2026-07-07
- **环境**：EKS `claude-sbx` + 2×c6g.metal（arm64）+ Firecracker driver（`SANDBOX_DRIVER=firecracker`）
- **测试脚本**：`scripts/warm_pool_e2e.sh`（本次新增）
- **结论**：暖池功能**首次真机跑通**（此前只有 moto mock 单测覆盖，从未真机验证）。过程中暴露并修复 **4 个真实 bug**，并得出一个关键结论——**暖池在当前精简 rootfs 上几乎无加速收益**（见 §4）。

## 1. 背景：为什么要做这次测试

暖池此前只有 `smoke_test.py` 的 moto mock 单测（`TestWarmPool`），验证的是 claim/replenish 的 DynamoDB 状态流转逻辑，**从未在真机上验证过 resume 出来的 VM 是否真能用、是否真的省时间**。`scripts/e2e_test.sh`（T1–T17）也没有任何暖池用例。这次专门补齐真机验证。

## 3. 暴露并修复的 4 个真实 Bug

这次测试的最大价值在于：**暖池在真机上一开始 100% 失败**（每次 `create` 都静默回退冷建），逐层排查修复了 4 个此前 mock 单测完全无法发现的 bug。

### Bug 1 — resume 的 tap 设备冲突（`node-agent/main.py`）
- **现象**：`create` 走暖池时 node-agent 报
  `Open tap device failed: Resource busy (os error 16). Invalid TUN/TAP Backend provided by fctapNN`。
- **根因**：
  1. `op_resume` 里 `_setup_tap` 在 `snapshot/load` **之后**执行；而 `snapshot/load(resume_vm=True)` 会立即恢复快照里保存的网络设备并打开宿主 tap，此时 tap 尚未就绪。
  2. `op_suspend`/`op_destroy` kill VMM 后**从不删除 tap 设备**，导致 tap 泄漏堆积（实测一台节点残留 11 个僵尸 `fctap*`）；tap_idx 复用时与残留设备冲突。
- **修复**：
  - `_setup_tap` 先幂等 `ip link del` 同名残留 tap 再新建；
  - `op_resume` 把 `_setup_tap` 移到 `snapshot/load` **之前**；
  - `op_suspend` kill VMM 后 `_teardown_tap` 释放 tap。
- **验证**：连续 claim-resume 从 0/3 成功 → 3/3 成功，resume 耗时 0.11–0.13s。

### Bug 2 — claim 迁移用了 `db.put`（条件写）导致必然失败（`sandbox-api/warm_pool.py`）
- **现象**：日志 `ConditionalCheckFailedException ... PutItem ... conditional request failed`；warm 计数减 1 但沙盒走了冷建（`snapshot_s3` 为空）。
- **根因**：`create_sandbox` 已先 `db.put` 写了一条 `id=real_id`（state=creating）占位记录；`claim()` 里 resume 成功后又 `db.put(new_record)`（同一 `real_id`），而 `db.put` 带 `attribute_not_exists(id)` 条件写 → 必然抛异常 → except 静默吞掉 → 删 warm → 回退冷建。**暖池形同虚设。**
- **修复**：迁移改用 `db.force_update(real_id, ...)` 覆盖占位记录。

### Bug 3 — `force_update` 的 `updated_at` 路径重叠（`sandbox-api/warm_pool.py`）
- **现象**：`Invalid UpdateExpression: Two document paths overlap: [updated_at]`。
- **根因**：迁移字段从 warm 记录继承了 `updated_at`，而 `force_update` 内部又固定 `SET updated_at=:now` → UpdateExpression 里 `updated_at` 出现两次。
- **修复**：迁移时排除 `updated_at`（连同 `id/pool_state/tenant_id/created_at`）。

### Bug 4 — warm_id/real_id 身份不一致导致 exec/vsock 失败（`firecracker.py` + `node-agent/main.py`）
- **现象**：claim 成功、`create` 返回 201，但 `exec` 报 `sandbox <real_id> not running`；修 id 后又报
  `VsockUnixBackend: Error binding to the host-side Unix socket: Address in use (os error 98)`。
- **根因**：`resume` 用 `warm_id` 注册 node-agent 的 `_VMS`，但沙盒对外身份是 `real_id`，后续 `exec`/`suspend` 按 `real_id` 路由 → 对不上。而快照把 vsock UDS 路径固化成 `{warm_id}/v.sock`，FC `snapshot/load` 会重新绑定该路径。
- **修复**：
  - `driver.resume` 增加 `snapshot_id` 参数：node-agent 用 `sandbox_id`(=real_id) 注册，但从 `snapshot_id`(=warm_id) 的快照/rootfs 路径恢复；
  - `op_resume` 清理并让 FC 绑定**快照来源目录**的 v.sock，resume 后在 `{real_id}/v.sock` 建 symlink 指向它，使 exec 的 vsock 主通道可达。

### 附带：可观测性缺失（已改进）
排查初期发现控制面 / node-agent **日志全空**——`claim` 的 except 静默吞异常，导致"暖池全回退冷建"时无任何线索。已把 `warm_pool.claim` 的 except 改为打印异常到 stderr（`flush=True`）。**这是这次能定位到 Bug 2/4 的关键。**

### 附带：resume 节点选择（`firecracker.py`）
`resume` 原本无条件 `_pick_node()` 重选节点。实测**跨节点 resume 需从 S3 下载整份内存镜像，耗时 ~78s**，远慢于冷建（0.17s），完全背离暖池初衷。已改为优先复用快照原节点（本地有快照，亚秒 resume），仅当原节点已死时才跨节点兜底。

## 4. 关键结论：暖池在当前 rootfs 上几乎无加速收益 ⚠️

修复后测得**底层纯耗时**（控制面内直接调 driver，排除 API/DynamoDB/port-forward 开销）：

| 路径 | 耗时 | 说明 |
|---|---|---|
| 纯 resume（暖池核心） | **0.130s** | snapshot/load + setup_tap |
| 纯 `driver.create`（冷建） | **0.169s** | rootfs CoW 复制 + FC boot |
| **加速比** | **仅 1.3x** | — |

端到端 API `create`：暖池 **1.43s** vs 冷建 **1.33s**——**基本无差异，暖池甚至因 S3/tap 开销略慢**。

**为什么与 README 宣称的"claim ≈ 7ms、冷建数量级差异"不符？**
- 测试用的是空白 **min-rootfs**：冷建的 `driver.create` 本身就极快（CoW 复制 + FC 微秒级 boot ≈ 0.17s），暖池没有多少可省的。
- 暖池的价值只在**冷建昂贵**时才体现：rootfs 巨大、boot 时要装依赖/起服务、guest 应用初始化慢（如预装 node_modules、预热 JVM/Python 运行时的真实业务镜像）。此时冷建可能要几秒~几十秒，而 resume 恒定 ~0.13s，才会有数量级差异。
- **`7ms` 这个数字来源存疑**（可能是极早期同机 diff 快照的测量），当前真机 Full 快照 resume 稳定在 **~0.13s**，不是 7ms。

**行动项**：README 的暖池宣称需要修正——把"claim ≈ 7ms"改为实测 `~0.13s`，并明确说明"加速收益取决于业务镜像的冷启动成本，空白 rootfs 场景收益有限"。

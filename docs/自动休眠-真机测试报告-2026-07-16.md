# 自动休眠 / 唤醒（auto-sleep / auto-wake）真机 e2e 测试报告

- **日期**：2026-07-16
- **环境**：EKS `claude-sbx` + 2×c6g.metal（arm64，kernel 6.1.174-amzn2023）+ Firecracker driver（`SANDBOX_DRIVER=firecracker`）+ 控制面 2 副本
- **测试脚本**：`scripts/autosleep_e2e.sh`（本次新增，A0~A5）
- **单测**：`sandbox-api/smoke_test.py` 的 `TestAutoSleep`（6 用例，moto mock），全套 48 用例通过
- **结论**：自动休眠 / 唤醒功能**首次真机跑通**。空闲沙盒自动进入独立状态 `slept`、网关请求透明唤醒回 `running`、活跃沙盒不误睡、手动 `suspended` 不被网关唤醒——**自动 vs 手动休眠的区分成立**。

---

## 1. 背景：为什么要做这个 feature

此前平台只有**手动** suspend/resume:沙盒创建后一直占节点 RAM,没人用也不会自己睡,要省钱得调用方主动调 `/suspend`。这与 fly.io Machines 的核心体验(没流量自动 `auto_stop`、来请求在网关 `auto_start`、对用户无感)有差距。

本次参照 fly.io 补齐这一环。基础件都已就位:`driver.py` 的 `ServiceSpec.autostop/autostart` 字段注释本就写着"字段占位,**唤醒代理下阶段实现**";FC 秒级快照已真机验证;`resolve_proxy_target` 是所有 `/s/` 流量的必经网关入口;`Reconciler`/`WarmPool` 给了 leader 门控 loop 的现成范式。

---

## 2. 设计要点

- **自动休眠用独立状态 `slept`**,与手动 `suspended` 严格区分。底层同一套 FC 快照,区别在:**只有 `slept` 会被网关请求自动唤醒**,手动 `suspended` 打到网关维持 409。
- **opt-in,默认关**:复用 Fly 语义的 `services[].autostop/autostart`(或 `meta.auto_sleep/auto_wake`)。不声明的沙盒行为完全不变。
- **活跃信号**:经网关 HTTP 流量 + `exec` 都刷新 `last_active_at`;热路径内存节流(`ACTIVITY_TOUCH_MIN_S`,默认 15s)防写放大。
- **后台扫描**:`sandbox-api/autosleep.py` 的 `AutoSleeper`,镜像 `warm_pool.py`,leader 门控(复用 reconcile/暖池同一 leader 锁);拿 lease 后**二次校验仍空闲**,防"扫描判定→加锁"之间刚来请求被误睡。
- **网关透明唤醒**:`resolve_proxy_target` 遇 `slept`+autostart → 调 `resume_sandbox`(改为接受 prev∈{suspended,slept})并轮询等其回 running 再转发。并发请求靠 lease 条件写互斥。
- **复用现有并发保护**:自动休眠走与手动 suspend 同一套 `lease + prev_state 条件写 + 失败回滚`,快照失败回滚 `running` 绝不静默丢数据。

## 3. 测试用例（A0–A5）

| 用例 | 验证点 | 结果 |
|---|---|---|
| **A0** | `GET /capabilities` 的 `suspend_resume=true`(前提) | ✅ PASS |
| **A1** | 创建 opt-in(autostop+autostart)沙盒(`image=web`,自起 :80)→ running | ✅ PASS |
| **A2** | 静置超过 idle → 自动进入 **`slept`**(≠ 手动 `suspended`);事件带 `reason=idle` | ✅ PASS |
| **A3** | 网关透明唤醒:`curl /s/{id}/80/` 首请求触发 resume → 回 `running`,`last_active_at` 刷新 | ✅ PASS(首请求 ~2.1s) |
| **A4** | 活跃不误睡:持续 exec 保活的沙盒,静置超过 idle 仍 `running` | ✅ PASS |
| **A5** | 手动/自动区分:手动 `/suspend` → `suspended`;网关 `curl` 不唤醒它(409,仍 `suspended`) | ✅ PASS |

> 测试期间集群 env 临时设 `AUTO_SLEEP_IDLE_S=30 AUTO_SLEEP_SCAN_S=15`(演示用),故 A2 静置约 30s 即入睡。代码默认 `AUTO_SLEEP_IDLE_S=300`(生产默认)。控制面 `ALLOW_UNAUTHENTICATED=1`、`NLB_HOSTNAME` 为空,`/s/` 反代由 control-plane 自身服务,port-forward 直打即测网关唤醒。

关键观测(A3):`slept` 沙盒首个 `/s/{id}/80/` 请求阻塞 ~2.1s(触发 resume + 等收敛),随后回 `running` 并正常转发 —— 与 fly.io "首请求略慢、之后无感"一致。

## 4. 踩坑与修复

1. **bash 3.2(macOS 自带)空数组 + `set -u`**:`"${AUTH[@]}"` 与 `"${AUTH[@]:-}"` 在未传 `--api-key`(空数组)时都报 unbound / 展开成空串坑 curl(`option : blank argument`)。正确写法 `${AUTH[@]+"${AUTH[@]}"}`(仅 set 时展开)。`warm_pool_e2e.sh` 因总传 API_KEY 未暴露此坑。
2. **port-forward 后 `sleep 3` 太短**:首个 `/capabilities` 空响应被误判 FAIL;改 `sleep 6`。
3. **验证入睡竞态**:`_ensure_awake` 轮询与后台扫描共用状态,靠 lease 条件写天然互斥,实测并发无重复 resume。

## 6. 配置速查（控制面 env,均可 `kubectl set env` 热更新）

| 变量 | 默认 | 含义 |
|---|---|---|
| `AUTO_SLEEP_ENABLED` | `1` | 总开关(仅作用于 opt-in 沙盒) |
| `AUTO_SLEEP_IDLE_S` | `300` | 空闲多久自动 sleep;实际入睡 = 该值 + 最多一个扫描周期 |
| `AUTO_SLEEP_SCAN_S` | `30` | 扫描间隔(leader 门控) |
| `AUTO_WAKE_TIMEOUT_S` | `30` | 网关唤醒等待上限 |
| `ACTIVITY_TOUCH_MIN_S` | `15` | 活跃时间写节流下限 |

> opt-in 方式:create 时 `services:[{"port":80,"autostop":true,"autostart":true}]`,或 `meta:{"auto_sleep":true,"auto_wake":true}`(不依赖端口声明)。Portal API Playground 创建表单勾选"自动休眠 / 唤醒"即自动带上。

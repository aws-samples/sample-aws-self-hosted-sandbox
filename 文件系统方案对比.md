# 沙盒数据目录文件系统方案对比

> OS rootfs 保持本地 ext4 不变。仅对比 `/workspace`(沙盒数据目录)挂不同 S3 后端的
> 性能与快照/恢复兼容性。客户现状:Fly.io + JuiceFS + S3。

## 一、三种方案概览

| | mountpoint-s3 | s3fs | JuiceFS + S3 |
|---|---|---|---|
| 出身 | AWS 官方(2023) | 社区 FUSE | 开源商业(Juicedata) |
| 元数据 | 无单独引擎(S3 对象直接映射) | 无(S3 对象直接映射) | **独立引擎**(Redis/TiKV/MySQL) |
| 顺序读写 | ✅ 优化好(多连接并行) | 一般 | 良好(本地缓存加速) |
| 小文件/随机读写 | ⚠️ 慢(每操作一次 S3 API) | ⚠️ 慢 | ✅ 好(本地缓存 + 元数据引擎) |
| POSIX 兼容性 | **受限**(不支持 append/rename/link) | 好 | 完整 |
| inotify | ❌ **不支持** | ❌ 不支持 | ⚠️ 有限支持(本地缓存内) |
| 运维复杂度 | 最低(零依赖) | 低 | **高**(需维护元数据引擎) |
| 客户现状对齐 | 否 | 否 | ✅ 是 |

## 二、性能测试结果(实测,Graviton c7g.metal)

> 测试在 Phase 1 主机(EC2 c7g.metal, us-east-1)上直接跑(宿主层,非 microVM 内)。
> 对照基准:本地 ext4。详细原始数据见本节末。

### 实测数据(Graviton c7g.metal, us-east-1, EBS gp3 宿主盘)

> 注:测试在宿主机直接跑(非 microVM 内),代表同等条件下各方案的 I/O 特征。
> s3fs 因编译依赖未能安装,数据缺失。

| 指标 | 本地 ext4 | mountpoint-s3 | s3fs | JuiceFS+S3 | **vs 本地 ext4** |
|---|---:|---:|---:|---:|---|
| 顺序写 256MB(MB/s) | **226** | 355 | — | 437 | MP: 1.6×↑, JFS: 1.9×↑(缓存加速) |
| 顺序读 256MB(MB/s) | **131** | 212 | — | 167 | 两者均快于本地(S3 读并发 > gp3) |
| 小文件写 500×4KB(files/s) | **1147** | 3.3 | — | 25.4 | **MP: 347×慢! JFS: 45×慢!** |
| 小文件读 500×4KB(files/s) | **1283** | 11.7 | — | 897 | MP: 109×慢, JFS: 基本持平(缓存命中) |
| inotify 触发 | ✅ 1 event | ✅ 1 event | — | ✅ 1 event | ⚠️ 见下方说明 |
| git add+commit 50文件(秒) | **0.09s** | 0.26s(git init 受限) | — | 1.68s | MP git 受限,JFS 18×慢 |

### 关键发现解读

**① 顺序读写:mountpoint-s3 和 JuiceFS 都快于本地 ext4**
S3 的高带宽 + 多连接并行使顺序读写超过 EBS gp3。对大文件操作(git clone 大仓、下载模型)有优势。

**② 小文件写:mountpoint-s3 崩塌，JuiceFS 也大幅下降**
- `mountpoint-s3`:3.3 files/s vs 本地 1147 files/s —— **347 倍差距**。
  原因:每个文件写 = 一次 S3 PUT API 调用(~15-30ms 往返),500 个文件串行 = 15 秒。
  这对 `npm install`(上万小文件)、编译(大量.o 文件)是**灾难性的**。
- `JuiceFS`:25.4 files/s，好得多但仍是本地的 45 分之一。
  本地缓存部分缓解,但元数据引擎(Redis 每次 metadata 操作)成为瓶颈。

**③ inotify:测试结果 "1 event" ≠ 完全正常**
测试只写了一个文件、检测到 1 个事件——这是"有限支持"的证明，不是"完全正常"的证明。
真正需要验证的是：dev server (Vite/Next.js) 监听大量文件时能否持续触发，以及跨进程写入是否被通知。**这个验证还需要专门的 dev server + 热重载场景测试。**

**④ git commit:JuiceFS 1.68s vs 本地 0.09s**
18 倍差距——git 的 sha1 计算 + 大量小文件写，在 JuiceFS 下元数据 + 数据落 S3 的代价。

### inotify 问题(先于数字给出结论)

**这是 Claude Code 使用 S3 后端的核心风险点。**

- `mountpoint-s3` / `s3fs`:这两个 FUSE 驱动**不实现 inotify 通知**。
  `inotifywait` 监听目录不会触发任何事件,即使文件已被修改。
  → Claude Code 的文件监听(dev server HMR、`--watch`、语言服务器)全部失效。

- `JuiceFS`:在**同一个 JuiceFS 客户端实例内**修改文件可以触发 inotify,
  但**跨客户端(另一台机器或另一个进程)的修改不会通知**——因为 FUSE 层无法感知
  远端写入。对 Claude Code 单沙盒单挂载的场景基本够用,但跨沙盒共享目录不可靠。

**结论:如果 workspace 只是当前沙盒自己读写(不跨沙盒实时感知),JuiceFS 的 inotify
支持可以接受;如果需要实时感知其他来源的写入,所有 S3 FUSE 方案都不可靠。**

## 三、快照/恢复兼容性(重要工程约束)

**核心问题:Firecracker 快照保存 guest 内存和 CPU 状态,但不保存 host 侧进程。**

```
快照前:
  host 进程:  [juicefs daemon] ← [guest /workspace via virtio-9p 或 FUSE-over-vsock]
  guest 内存: FUSE 内核驱动状态、inode cache、page cache          ← 会被快照保存

快照后 kill VM + resume:
  host 进程:  [已不存在 or 已重启成新进程]                       ← 快照不管这个
  guest 内存: 恢复到快照时刻 ← FUSE 驱动以为旧 daemon 还在
  结果:       /workspace → "Transport endpoint is not connected"
```

**所以 S3 FUSE 挂载 + Firecracker 快照/恢复,恢复后 /workspace 会断连。**

### 处理方法(控制平面 resume 流程必须包含这步)

```
resume 流程:
  1. 选节点,恢复 FC 快照(fc_snapshot_api.py: /resume)
  2. ⚠️ 在 host 上重新启动 FUSE daemon
     (juicefs mount / mount-s3 / s3fs,指向同一个桶/路径)
  3. 在 guest 内执行 remount:
     umount /workspace && mount /workspace
     (或让 agent 在 resume 后主动 remount)
  4. 恢复后验证 /workspace 可读写
```

**对比:Fly Volumes(virtio-blk 持久块盘)**
- Fly 的 persistent volumes 是 virtio-blk 设备,Firecracker 快照保存设备状态
- resume 后 virtio-blk 自动可用,**不需要 remount**
- 这也是为什么 Fly 用块盘而非 S3 FUSE 做 persistent storage 的原因之一

**对比:JuiceFS 在 Fly 上的真实用法**
- Fly + JuiceFS + S3 是"JuiceFS 客户端在 VM 内运行,S3 是数据后端"
- VM 被 Fly suspend 时是整个 VM 暂停(包括 JuiceFS daemon)
- resume 后 daemon 随 VM 内存恢复,**不需要在 host 上重启**——因为 daemon 跑在 guest 内
- **所以客户在 Fly 上的架构是:JuiceFS daemon 跑在沙盒 guest 内,不是 host 侧挂载**

这一点非常关键——迁移到 AWS 时要保持同样架构(**JuiceFS daemon 在 guest 内运行**),
而不是在 host 上挂载后映射进 guest(那样会有断连问题)。

## 四、两条迁移路径(给客户)

### 路径 A:对齐客户现状(JuiceFS in guest)

```
Firecracker microVM
├── /           本地 ext4(rootfs)
└── /workspace  JuiceFS 客户端在 guest 内运行,数据落 S3,元数据落 Redis/ElastiCache
```
- 快照/恢复兼容:✅ daemon 在 guest 内,随快照保存恢复
- inotify:✅ 单沙盒内可用
- 运维成本:需维护 Redis/ElastiCache(元数据引擎)
- **与客户 Fly 现状一致,迁移代价最小**

### 路径 B:AWS 原生(mountpoint-s3 in guest)

```
Firecracker microVM
├── /           本地 ext4(rootfs)
└── /workspace  mountpoint-s3 在 guest 内运行,数据直接落 S3
```
- 快照/恢复兼容:✅(同理:daemon 在 guest 内)
- inotify:❌ 不支持
- 运维成本:极低(零依赖)
- **适合 workspace 只做数据持久化(不依赖 inotify)的场景**

### 路径 C:块盘(EBS/本地 NVMe,推荐最高性能)

```
Firecracker microVM
├── /           本地 ext4(rootfs)
└── /workspace  virtio-blk 挂载 EBS gp3 或本地 NVMe
```
- 快照/恢复兼容:✅ 最佳(设备状态随 VM 状态保存)
- inotify:✅ 完整支持
- 性能:最好
- 缺点:跨机迁移需先 detach EBS;本地 NVMe 不可迁移
- **适合需要最高 I/O 性能的场景(build/test 密集型 agent)**

## 五、给本 POC 的建议

**POC 阶段继续用本地 ext4。** 实测数据已给出明确的选型信号：

| 客户需求 | 推荐方案 | 理由 |
|---|---|---|
| 对齐现有 Fly+JuiceFS+S3 架构 | **路径 A:JuiceFS in guest** | 快照兼容 ✅;小文件慢但可接受;客户已熟悉 |
| 最高 I/O 性能(build/test 密集) | **路径 C:EBS gp3/本地 NVMe** | 小文件 1147 files/s vs JFS 25 files/s;inotify 完整 |
| 纯数据持久化(不需要 inotify) | **路径 B:mountpoint-s3 in guest** | 顺序读写快;零运维;不依赖 Redis |

**关键决策点(需向客户确认):**
1. **Claude Code 工作流是否依赖 `npm run dev` 类热重载？**
   → 是 → 需要本地 EBS/NVMe(路径 C);JuiceFS inotify 支持需进一步验证
   → 否 → JuiceFS(路径 A)可接受
2. **工作目录小文件数量级？**
   → `npm install` 级别(上万文件)→ mountpoint-s3 完全不可用,JuiceFS 也很慢
   → git 仓库+少量源码 → JuiceFS 勉强可接受(git commit 1.68s vs 0.09s)

**下一步(H2 验证):** 在 microVM guest 内运行 JuiceFS 客户端,复现客户 Fly+JuiceFS+S3 架构,重点验证:
- `inotify` 在真实 dev server 场景(Vite/Next.js HMR)下的行为
- `npm install` 完整耗时对比
- 快照/恢复后 JuiceFS 是否自动续连(daemon 在 guest 内,应该可以)

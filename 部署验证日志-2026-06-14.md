### Step 3 — Kata 安装 ⚠️ 重大发现：节点被 EKS 健康检查替换（与既有认知不同）
严格逐字执行 README Step 3（nfd repo → dependency build → helm install → rollout status 1200s → wait node 1200s）。

实测时间线（节点 ip-10-0-101-105）：
- T+0: helm install kata-deploy 成功，20 个 runtimeclass（含 kata-qemu）约 6 分钟内创建完成
- T+90s: 节点进入 NotReady（`systemctl restart containerd` 触发，符合 README 警告）
- T+12min: 节点 Ready=Unknown，**EC2 实例状态检查 impaired/impaired**
- T+15min: **关键差异** —— EKS 托管节点组的健康检查判定该实例不健康，**自动起了一个替换节点 ip-10-0-101-49（ASG max=2 允许），并把老节点 cordon（SchedulingDisabled）**
- T+16-18min: 老节点 105 反而自愈恢复 Ready（但保持 cordoned）；新节点 49 起来后 kata-deploy pod Running 且**稳定 4+ 分钟未再崩溃**
- 节点组最终 desired=1，health issues 为空，会逐步 drain+terminate 老节点回到 1 节点

**与既有记忆（kata-deploy-c6g-metal-node-crash）的差异**：
之前记录是「节点最终自愈，不要 reboot/terminate」。本次实测发现 **EKS 托管节点组（min=1/max=2）会主动判定 impaired 节点不健康并触发节点替换**——这不是用户操作，是 EKS 自动行为。结果是部署中途会短暂出现 2 个 metal 节点、老节点被 cordon。这对 README 有两点影响：
  1. README 警告说「切勿手动 reboot/terminate 节点」是对的，但**没说 EKS 自己会替换节点**。用户照着 `kubectl wait node --all` 等，会看到节点数从 1→2、老节点 SchedulingDisabled，可能误以为出错。
  2. 替换后新节点重新跑一遍 kata-deploy（又一次 containerd 重启），但本次实测新节点没有再 impaired，最终收敛。**建议 README 补充**：装 kata 后若看到节点被替换/出现第二个 metal 节点属正常，等节点组收敛到 desired 即可；不要干预。
  3. phase3 节点组 `max_size = 2` 给了替换余量；若 max=1 则无法自动替换，可能卡更久。这点 max=2 的设置反而是有利的，可在文档说明。

### Step 3 — Kata 安装 ★★★ 根因定位（决定性证据）

**之前以为「containerd 重启慢」是错的。真实根因：节点整体 hang ~12 分钟后重新开机。**

在稳定的节点 225（i-0e8d2bf8baf2dad05）上用 SSM 抓 journalctl + uptime，时间线：
- `07:36:41` kubelet 首次启动，节点 Ready
- `07:37:27` kata-deploy 第一次 `systemctl restart containerd` —— **仅耗时 200ms**（Stopping→Started），containerd 重启本身根本不慢
- 但留下孤儿进程：`containerd.service: Unit process NNNN (containerd-shim) remains running after unit stopped`
- `07:40:45` kubelet 被 Stopped
- `07:53:05` **`uptime -s` 显示系统在此时重新开机** ← 节点真的 **hang/重启了约 12 分钟**
- `07:54:13` 重启后 kubelet 才重新起来，期间 aws-cni 50051 connection refused，pod sandbox 无法创建

**结论**：c6g.metal + AL2023 + containerd 2.2.3 上，kata-deploy 触发的 containerd 重启会让**整个裸金属节点 hang 约 12 分钟后才重新开机**（不是 containerd 慢，是节点级挂死）。这期间 EC2 instance/system reachability 检查失败 → 托管节点组的 ASG（max=2）ReplaceUnhealthy 在节点自愈前就把它当 unhealthy 替换 → 新节点装 kata 又重复 → **节点替换循环**（实测 105→49→225 三次替换）。

ASG 活动历史证实：07:22、07:36 两次 "EC2 instance status checks failure" 触发替换。手动 `suspend-processes ReplaceUnhealthy HealthCheck` 后不再有新替换，225 最终稳定 Ready（约 20 分钟自愈）。kata 最终可用（20 个 runtimeclass，kata-qemu 在）。

**对 README 的影响（严重）**：当前 README Step 3 把节点 NotReady 当"正常等待"，但实际是 ASG 替换循环，且 README 没提 ASG 会替换节点。即便耐心等，托管节点组 max=2 也可能反复替换。这是阻断性体验问题。

### 方案 A 验证（用户指定 Karpenter EC2NodeClass.userData）
核心思路：把 kata 安装 + containerd 重启**前置到节点 bootstrap（kubelet 向 EKS 注册之前）**，则节点 hang/重启发生在 EKS 尚未看到该节点时，不触发健康检查替换。

**用户给的方案 A 原文需要修正的点（实测发现）**：
1. **artifact URL 错误**：`kata-static-3.31.0-arm64.tar.xz` 返回 404。真实文件是 `kata-static-3.31.0-arm64.tar.**zst**`（zstd 压缩，660MB）。解压要用 `tar --use-compress-program=unzstd -xf`，不是 `-xJf`。
2. **containerd 配置路径用的是旧版**：方案 A 写 `[plugins."io.containerd.grpc.v1.cri".containerd.runtimes.kata-qemu]`，但 AL2023 的 containerd 2.2.3 用 **v2 新路径** `[plugins."io.containerd.cri.v1.runtime".containerd.runtimes.kata-qemu]`。用错路径 containerd 不认 kata runtime。
3. 真实 kata-deploy 写的 drop-in（`/opt/kata/containerd/config.d/kata-deploy.toml`）关键最小子集（仅 kata-qemu）：
   - `runtime_type = "io.containerd.kata-qemu.v2"`
   - `runtime_path = "/opt/kata/bin/containerd-shim-kata-v2"`
   - `ConfigPath = "/opt/kata/share/defaults/kata-containers/runtimes/qemu/configuration-qemu.toml"`（注意 3.31 的路径含 `runtimes/qemu/` 子目录）
   - base config.toml 加 `imports = ["/opt/kata/containerd/config.d/kata-deploy.toml"]`
4. kata-static 解压根目录是 `/opt/kata`（bin/qemu-system-aarch64、share/defaults/... 都在其下）。

### 方案 A 验证结果 ★★★ 成功（决定性）

用 Karpenter kata-metal EC2NodeClass.userData 预装 kata，起新 c6g.metal 节点 ip-10-0-102-223：
- `08:46:00` 开机 → `08:46:13` containerd 启动 → `08:46:14` containerd 重启(200ms)+kubelet 启动 → `08:47:05` 节点 Ready
- **从注册到 Ready 仅约 30-60 秒，全程无 NotReady 抖动、无 hang、无 ASG 替换**
- containerd 重启总次数仅 2（boot 一次 + UserData 一次），都在开机后 14 秒内完成
- probe pod 用 kata-qemu 成功 Running，guest kernel 6.18.28 ≠ node 6.1.172，nproc 隔离 → 真 microVM

**为什么方案 A 有效（根因闭环）**：
kata-deploy 路线在**已运行 kubelet + 多个容器**的节点上 `systemctl restart containerd`，留下孤儿 containerd-shim，导致 c6g.metal 节点整体 hang ~12 分钟后重新开机。而方案 A 在 **bootstrap 阶段（容器为空、kubelet 尚未由 nodeadm 启动）** 重启 containerd，此时重启是瞬时的（200ms），且 EKS 还没看到该节点，不触发健康检查替换。

**实测修正后的可用 UserData**（已存 /tmp/karpenter-kata-metal.yaml）关键点：
1. artifact: `kata-static-3.31.0-arm64.tar.zst`（zst 不是 xz），`tar --use-compress-program=unzstd -xf ... -C /`
2. containerd v2 drop-in 路径 `io.containerd.cri.v1.runtime`，ConfigPath=`/opt/kata/share/defaults/kata-containers/configuration-qemu.toml`
3. **NodePool template.metadata.labels 必须加 `katacontainers.io/kata-runtime: "true"`** —— 否则 kata-qemu RuntimeClass 的 nodeSelector 让 Karpenter 认为 NodePool 不兼容、拒绝起节点（实测踩坑）。
4. 方案 A 下 **不应再装 kata-deploy DaemonSet**（避免在 Karpenter 节点上重复触发 containerd hang）；但需要保留/单独创建 kata-qemu RuntimeClass（本次验证里 RuntimeClass 是 kata-deploy 建的，复用了）。

### Step 8-9 — 端到端测试

**port-forward 模式：✅ ALL TESTS PASSED**（exit=0）

- T10 suspend/resume skip（kata driver 不支持，符合预期）。

**生产 Ingress + --resolve 模式：❌ 开箱不通（README Step 9 主推的模式）**
现象：`scripts/e2e_test.sh --api-url http://api.sbx.example.com --resolve ...` 在 T1 即 curl exit 52（empty reply, HTTP:000）。
逐层诊断：
1. 集群内 ClusterIP 访问 ingress→控制面 = **HTTP 200**（控制面 + Ingress 路由本身正常）
2. NLB 三个 IP TCP:80 都能连（nc 成功），但 HTTP empty reply
3. **直接打节点公网 IP:31827（NodePort，绕过 NLB）也是 HTTP:000** ← 关键
4. 目标组 `preserve_client_ip.enabled=true` + `externalTrafficPolicy=Cluster`，target type=instance

**根因**：本架构用 ingress-nginx 的 **in-tree（cloud-provider）NLB**，target type=instance + preserve_client_ip=true。外部流量到 NodePort 后 kube-proxy 跨节点 SNAT 转发 + 保留客户端 IP 导致回包非对称路由，连接挂死。叠加 Karpenter kata-metal 节点（带 kata-dedicated taint、无 ingress pod、跨 AZ）被混入 NLB 目标组（一直 unhealthy/initial），NLB 默认 cross-zone 关闭，进一步放大问题。

**对 README 的影响（严重）**：README Step 9 把「生产 Ingress + --resolve」列为**推荐**的 e2e 验证方式，但在当前 Terraform/Helm 装出来的架构里**开箱不通**。建议：
  - 要么在文档明确：外部 Ingress 访问需额外装 **AWS Load Balancer Controller**（target type=ip，直接指向 pod，绕过 NodePort/SNAT 问题）；
  - 要么 ingress-nginx 安装时加注解 `service.beta.kubernetes.io/aws-load-balancer-cross-zone-load-balancing-enabled=true` 并把 e2e 默认验证方式改为 port-forward；
  - 并说明 Karpenter kata-metal taint 节点不应进 ingress NLB 目标组（可用 `aws-load-balancer-target-node-labels` 限定只注册系统节点）。
  - 我实测中途加了 cross-zone 注解 + 试了 Local/Cluster 策略均未通，确认非单一注解可解，需要 LB Controller 或 NodePort 数据面排查。

# 沙盒快照 + 存储架构:Snapshot 包含什么、数据放哪、怎么跨机

> 回答:"数据挂到 S3 后 snapshot 是否还包含它?要不要学 E2B 的 local+异步S3?是不是干脆全放 JuiceFS?"
> 核心结论先行:**这三件事是不同层,不是二选一。跨机靠"复制 snapshot 到 S3",不是"把数据放 S3 挂载"。**

## 一、Firecracker Snapshot 到底包含什么

```
Snapshot = ┌─ vm.mem      (guest 全部内存页)
           └─ vm.snapshot (vCPU 状态 + 设备状态)
```

- **不包含磁盘内容。** 磁盘(block device)是 Firecracker 之外的独立文件(我们 demo 的 `rootfs.ext4`)。
  snapshot 只记录"挂了哪个磁盘文件在什么路径",**不复制磁盘数据**。

### ⚠️ 一致性铁律(整套架构的地基)

> 内存快照里**包含磁盘的 page cache、文件系统元数据、in-flight 写**。
> 所以 **mem 快照 与 它当时挂的那块 disk 必须配对** ——
> resume 内存时,那块磁盘必须是快照那一刻的样子。否则内存里缓存的磁盘视图和实际磁盘对不上 → **文件系统损坏**。

这条铁律决定了:**snapshot 的最小单元是 `mem + disk` 一对**,不能拆开、不能各自漂移。

## 二、数据挂到 S3,在不在 snapshot 里?

**数据本身不在,但有隐藏陷阱。** 分两种:

| 数据放哪 | 磁盘数据在 snapshot? | 跨机 resume 时要处理 |
|---|---|---|
| 本地块设备(rootfs.ext4 / EBS / NVMe) | 否,但**内存里有它的 page cache** | 必须把**磁盘文件和内存快照一起**搬到 B 机,且配对一致(铁律) |
| S3 网络挂载(JuiceFS / mountpoint) | 否,数据在 S3 | 数据天然在 S3✓,但**内存里有 FUSE client 状态 + 到 S3/Redis 的 TCP 连接** |

**第二行的陷阱:** S3 挂载的*数据*在 S3,但 microVM 内存里还冻着 **FUSE 客户端运行态 + 网络连接**。snapshot 能冻结内存状态,但 **TCP 连接 / open fd 不能跨 resume 存活**:
- FUSE daemon 在 **guest 内**(Fly+JuiceFS 模式):daemon 随内存恢复,但连接断,需**重连**(JuiceFS 能重连,但正在写的文件可能损坏)。
- FUSE daemon 在 **host**:它不在 snapshot 里,resume 后 `/workspace` → `Transport endpoint not connected`,必须 remount。

## 三、为什么只靠本地 snapshot 跨机就废

我们之前 demo 的 snapshot/resume 之所以成功,是因为**同机 resume——`rootfs.ext4` 还在本地**,而且我们的 snapshot **只有 vm.mem + vm.snapshot,没含 rootfs**。
换到 B 机 resume,B 机没有那个 `rootfs.ext4` → **直接失败**。

→ 这正是 E2B 要做 "local + 异步 S3" 的根本原因。

## 四、E2B 的 local + 异步 S3 = 三层存储模型

E2B 的 snapshot 单元 = **memfile + rootfs 一对**(满足一致性铁律),然后:

```
S3       = 真相源(durable, 跨机可达)        ← 异步同步一份,任意 host 可拉回
本地NVMe = 热缓存(同机命中极快)             ← 同机 resume 毫秒级(我们实测过)
内存     = 运行态(snapshot 捕获)
```

- 存本地 = 同机 resume 快;
- 异步同步 S3 = 持久 + 可移植,A 机 suspend → B 机能拉回来 resume。

## 五、是不是干脆全放 JuiceFS/S3?—— 不,关键认知纠正

**Snapshot 和 JuiceFS 不是二选一,是不同层、解决不同问题。JuiceFS 解决不了你真正的痛点。**

1. **JuiceFS 解决不了内存的跨机问题。**
   resume 一个*运行中*沙盒,仍需 memory snapshot,而 memory snapshot 仍需 local+S3 复制才能跨机。
   JuiceFS 只改变"workspace 磁盘在不在 snapshot 里",**完全没省掉 E2B 那套 memory 快照机制**。

2. **热数据放 JuiceFS,性能崩(本仓库实测)。**
   | | 本地 | JuiceFS+S3 | mountpoint-s3 |
   |---|---|---|---|
   | 小文件写 | 1147 files/s | 25 files/s(45×慢) | 3.3 files/s(347×慢) |
   Claude Code 跑 `npm install`/编译/git = 上万小文件 → 体验杀手。

3. **一致性更脆。** JuiceFS 是"活的远程挂载",内存快照与 S3 数据会漂移(别的进程也能改),
   snapshot 前要 quiesce、resume 后要 revalidate,比块设备 snapshot 脆弱得多。

## 六、推荐架构:三层,各放各处

```
┌─ microVM ───────────────────────────────────────┐
│  /           OS rootfs(Node+Claude Code+工具链)   │ ① 只读共享基础镜像
│              每台 host 拉一次, 所有沙盒 CoW 共享     │   内容寻址, 本地NVMe缓存, 源在 S3/ECR
│                                                   │   不需 per-sandbox 快照
│  /workspace  热工作集(repo / 编辑中的文件)          │ ② 块设备(CoW overlay), 本地 NVMe
│              随 snapshot 一对捕获                   │   snapshot(mem+disk) → 本地 + 异步S3
│                                                   │   = E2B/Fly 模式, 满足一致性铁律
│  /data       冷/大/跨沙盒共享数据(可选)             │ ③ JuiceFS/mountpoint in-guest
│              直接挂 S3, 不进 snapshot               │   resume 后重连; 绝不放热数据
└───────────────────────────────────────────────────┘
```

| 层 | 内容 | 存储 | 是否进 snapshot | 跨机 |
|---|---|---|---|---|
| ① OS rootfs | 工具链/不可变 | 内容寻址基础镜像, 本地缓存, 源 S3/ECR | 否(共享只读) | 每台 host 各拉一份 |
| ② 热工作集 | repo, 编辑文件 | 本地 NVMe 块设备 | **是(mem+disk 一对)** | snapshot 复制到 S3 → B 机拉回 |
| ③ 冷/共享数据 | 大文件, 跨沙盒共享 | JuiceFS/mountpoint(guest 内挂 S3) | 否 | 天然在 S3, resume 后重连 |

## 七、跨机 resume 流程(E2B 模式)

```
A 机 suspend:
  1. PATCH /vm Paused
  2. snapshot/create → vm.mem + vm.snapshot(+ 配套 rootfs/workspace 块设备)
  3. 存本地 NVMe(热)
  4. 异步上传 [vm.mem + vm.snapshot + workspace-disk] 三件套到 S3(同一 sandbox 前缀, 保证配对)
  5. kill VMM, 释放 RAM

B 机 resume:
  1. 从 S3 拉回 [vm.mem + vm.snapshot + workspace-disk] 到 B 机本地
  2. 起新 firecracker, snapshot/load(指向本地拉回的 mem + disk)
  3. resume_vm
  4. 重建 guest 网络(tap/IP)、重同步时钟(NTP)、丢弃旧 vsock
  5. 若有 ③ 层 JuiceFS/mountpoint:resume 后在 guest 内重连
```

**关键:第 4 步三件套必须配对**(同一 snapshot 时刻的 mem + disk),否则违反一致性铁律 → 文件系统损坏。

## 八、一句话决策框架

> **"运行中沙盒的暂停/恢复/跨机" = E2B 的 snapshot(mem+disk)local+异步S3,逃不掉,也不该用 JuiceFS 替代。**
> **JuiceFS/mountpoint 只用于"独立于沙盒生命周期、跨沙盒共享或太大不值得进 snapshot 的冷数据",且挂在 guest 内、不放热数据。**

最初的纠结("是不是全放 S3")源于把两件事当成一件:
**跨机靠"把 snapshot 复制到 S3",不是"把数据放到 S3 挂载"。**
前者(E2B)一次解决内存+磁盘+一致性;后者(JuiceFS)只解决一类数据存放,还带来性能与一致性代价。

### 跨机 resume 实测记录(2026-06-12, c7g.metal)

模拟 A 机 → B 机迁移(同一台机器不同路径模拟):

```
[A机] 创建沙盒,磁盘写标记 saved-on-A-xxx,内存计数 HB=3
[A机] suspend → 三件套: rootfs.ext4=8.0G  vm.mem=2.0G  vm.snapshot=11K
搬迁:  三件套 → 暂存区(模拟上传 S3) → 彻底删除 A 机(进程+文件全删)
[B机] 从暂存区拉回到【相同路径】→ snapshot/load
结果:  HB 从 3 续到 4,5,6 ✓   磁盘标记 saved-on-A 完整 ✓   resume 9.8ms
```

**✅ 证明铁律:** A 机文件全删后,B 机仅凭"S3 拉回的 mem+snapshot+rootfs 三件套"
独立 resume 成功,内存状态(HB 计数)与磁盘数据(marker)都精确续上。

### ⚠️ 实测踩到的关键坑:Firecracker snapshot 记录磁盘的【绝对路径】

测试中先犯了一个错,反而揭示了最重要的工程细节:

> `snapshot/create` 会把 backing file 的**原始绝对路径**写进 snapshot。
> `snapshot/load` 时 Firecracker **按这个原始路径去找 rootfs**,**不支持在 load 时覆盖 drive 路径**。

- ❌ B 机把 rootfs 放到**不同路径**(如 `/opt/machineB/`)→ load 报
  `No such file or directory /opt/machineA/rootfs.ext4` → 失败。
- ✅ B 机必须把 rootfs 放到**与 A 机完全相同的路径** → load 成功。

**生产含义:** 跨机迁移时,所有 host 必须用**统一的沙盒路径约定**(如每个沙盒固定挂在
`/var/lib/sandboxes/<sandbox-id>/rootfs.ext4`),B 机从 S3 拉回后放到同一路径再 load。
这是 E2B/Fly 跨机迁移能成立的隐含前提——**路径必须可重建且一致**。

> (注:`mem_file_path` 可在 load 时通过 `mem_backend.backend_path` 显式指定,不受此限;
> 受限的是 **block device 的 backing file 路径**,它锁定在 snapshot 内。)

# 沙盒状态存储设计:需不需要数据库?

> 回答"我需不需要一个数据库来存沙盒状态信息"。结论先行:**取决于路线**——
> Kata-on-EKS 初期不需要(etcd 当状态库);裸 Firecracker 必须有(自己存映射)。

## 一、核心原则:能否从运行时"现查"出来

数据库的本质作用 = 存**没法从运行时实时查出来**的信息。先把沙盒状态分两类:

| 类型 | 例子 | 能否现查 | 存哪 |
|---|---|---|---|
| **运行时状态** | 在不在、Running/Pending、哪个节点、IP、端口、哪个 VMM 进程 | ✅ 能 | K8s etcd(Kata)/ Firecracker 进程(裸 FC) |
| **业务元数据** | 哪个租户、创建人、创建时间、用途、计费时长、快照存哪、自定义标签 | ❌ 不能 | 需要持久层 |

→ 运行时状态有"真相源"(source of truth),不该再存一份(会不一致);业务元数据必须显式存。

## 二、路线 A:Kata-on-EKS —— 初期不需要专门数据库

**etcd(K8s 自带)已经是你的状态库。** 沙盒 = Pod,其状态/节点/标签/注解全在 etcd,`kubectl`/K8s API 就是查询接口。我们的 `sandbox-api/app.py` 即如此:**完全无状态**,每次 `GET /sandboxes` 实时查 etcd,不存任何东西。

**业务元数据塞进 Pod 的 labels / annotations:**
```yaml
metadata:
  labels:      { sandboxId: "234049", tenant: "acme", plan: "pro" }   # 可被 label selector 查询
  annotations: { created-by: "user-123", purpose: "code-review",
                 snapshot-s3: "s3://bucket/234049" }                  # 任意 KV,不可选择器查
```
这样连业务信息都在 etcd,**依然零外部数据库**。

**限制(超出就要加 DB):**
- label:键/值 ≤ 63 字符、值受限字符集;annotation:单个 Pod 所有注解合计 ≤ 256KB。
- etcd **删了就没了**:Pod 一删,历史无处可查。
- etcd 不是查询/报表数据库:做不了"上月各租户用量聚合"这类。

**何时给 Kata 路线加数据库(作旁路审计/计费层,不是真相源):**
1. 要**历史/审计**(沙盒销毁后仍要查谁在何时创建了多少)
2. 要**计费聚合 / 报表**(按运行时长计费)
3. 要**跨集群**统一视图
4. 元数据结构复杂,labels/annotations 存不下

> 关键:即便加了 DB,运行时状态仍以 etcd 为准;DB 是旁路记录(通过 watch/事件写入),不要让 DB 和 etcd 互相打架。

## 三、路线 B:裸 Firecracker —— 必须有数据库

这条路线没有 etcd,**你就是调度器**。"沙盒 ↔ 在哪台机器 ↔ 哪个 FC 进程 ↔ 快照文件在哪"这套映射**只能你自己存**。Fly / E2B 都有这层(E2B 用 Postgres,大用量分析另配 ClickHouse)。

**没有这张表,一个被快照挂起、进程已销毁的沙盒,你根本不知道它的快照在哪、该在哪台机器恢复**——这正是"挂起后怎么找回来"缺的那块。

最小核心表(概念):
```sql
CREATE TABLE sandboxes (
  id            TEXT PRIMARY KEY,
  tenant        TEXT NOT NULL,
  state         TEXT NOT NULL,           -- creating|running|suspended|deleted
  host          TEXT,                    -- 在哪台 .metal(running 时)
  fc_pid        INT,                     -- Firecracker 进程(running 时)
  guest_ip      TEXT,
  rootfs_path   TEXT,
  snapshot_path TEXT,                    -- 本地 NVMe / s3://...(suspended 时)
  cpu, mem_mib  INT,
  created_at, updated_at TIMESTAMPTZ,
  meta          JSONB                    -- 任意业务字段
);
```
这就是 Fly/E2B 自研控制平面里"重"的那部分之一。

## 四、两条路线对比

| | Kata-on-EKS | 裸 Firecracker |
|---|---|---|
| 运行时状态真相源 | **etcd**(白送) | **你的数据库**(自建) |
| 初期是否需要 DB | 否(元数据塞 labels/annotations) | **是(必须)** |
| 后期加 DB 的理由 | 审计/计费/报表(旁路) | 本来就是核心 |
| 一致性风险 | 低(单一真相源 etcd) | 需自己保证 DB 与实际进程/快照一致(对账) |

## 五、给本 POC 的建议(当前主推 EKS+Kata)

- **现在保持无状态**:`app.py` 用 etcd 查、元数据用 labels/annotations,**短期不建数据库**。这是对的,别过早引入。
- **触发加数据库的三个信号**(出现任一即上,推荐 **RDS Postgres**,对应文档 E2B 选型):
  1. **上快照/挂起** —— "sandbox ↔ snapshot ↔ 位置"映射 etcd 存不优雅,需 DB(这也是裸 FC 路线本就需要 DB 的同款理由);
  2. **上计费** —— 按运行时长计费要历史记录;
  3. **多租户审计** —— 谁在何时创建/销毁了什么。
- **落地形态**:加 DB 时仍让 etcd 当运行时真相源,DB 通过 K8s informer/watch 异步落历史与计费事件(旁路),避免双写不一致。

> 一句话:**Kata 路线"按需、旁路";裸 FC 路线"必需、核心"。** 你现在选 Kata,所以先不建;等上快照或计费时再加一个 Postgres 做旁路审计/映射。
